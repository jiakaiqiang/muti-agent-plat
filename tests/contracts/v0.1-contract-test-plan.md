# v0.1 Contract Test Plan

## API Contract

### Sessions

- `POST /api/sessions` accepts `input`, optional `agentIds`, `projectId`,
  `tokenBudget`, and `knowledgeBaseIds`.
- Response shape is `{ data: { session, firstEvent }, requestId }`.
- Created session includes `ownerId`, `workspaceId`, `participatingAgentIds`,
  `tokenUsed`, `createdAt`, and `updatedAt`.
- `firstEvent` is a valid `CollaborationEvent` with
  `metadata.schemaVersion = "0.1"`.
- `GET /api/sessions/:sessionId` returns the same session identity and current
  status.
- `POST /api/sessions/:sessionId/messages` persists a `user_message` event
  before routing and returns a `handlingPlan` when sent during `EXECUTING`.
- When an `EXECUTING` user message returns `handlingPlan.shouldPause = true`,
  the session transitions to `WAIT_USER_DECISION` and emits a high-priority
  `session_status_changed` event.
- Session control endpoints reject unsupported external status transitions with
  `INVALID_SESSION_TRANSITION`.

### Briefs

- `GET /api/sessions/:sessionId/briefs` returns ordered brief versions.
- `POST /api/sessions/:sessionId/briefs/:briefId/confirm` marks
  `confirmedByUser = true`, sets `confirmedAt`, emits `brief_confirmed`, and
  creates executable `AgentTask` records.
- Confirming an already confirmed brief returns `BRIEF_ALREADY_CONFIRMED`.
- Rejecting a brief emits `brief_rejected` and moves the session to
  `REVISING_BRIEF` or an equivalent contract-defined status.

### Events

- `GET /api/sessions/:sessionId/events` returns `PageResponse<CollaborationEvent>`.
- `afterEventId` returns only later events and does not duplicate the referenced
  event.
- Reconnecting `GET /api/sessions/:sessionId/events/stream` can receive later
  live events created after the new stream is opened.
- `GET /api/sessions/:sessionId/events/stream` emits SSE messages as:

```text
event: collaboration-event
id: <eventId>
data: <CollaborationEvent JSON>
```

- Heartbeats use `event: heartbeat`.

### Knowledge

- Knowledge base creation returns contract fields including `scope`,
  `visibility`, and `embeddingModel`.
- Document creation transitions toward `ready` or reports `failed`.
- Search returns `chunks: RagMatchedChunk[]`.
- A successful runtime retrieval is also represented as a `rag_retrieved`
  collaboration event.

### Memory

- `POST /api/sessions/:sessionId/memories` creates a session-scoped memory item.
- `GET /api/sessions/:sessionId/memories?q=` returns relevant memories.
- `preference_input` user messages can create `long_term_candidate` memories.
- Runtime Context Pack includes relevant memories and emits `memory_used` when
  memory is injected.

### Artifacts

- `GET /api/sessions/:sessionId/artifacts` includes artifacts created during
  dry-run, review, and delivery.
- `GET /api/artifacts/:artifactId` returns the same artifact with
  `sessionId` and `metadata`.

### Debug

- `GET /api/sessions/:sessionId/debug/context-packs` returns runtime Context
  Pack snapshots.
- `GET /api/sessions/:sessionId/debug/runtime-invocations` returns invocation
  status, usage, and Context Pack summary.
- `GET /api/sessions/:sessionId/debug/rag-retrievals` returns RAG retrievals
  derived from events.
- `GET /api/sessions/:sessionId/debug/token-usage` returns session-level token
  usage summary.

### Ops

- `GET /api/health` returns service status, version, and timestamp.
- `GET /api/ops/queues` returns the configured BullMQ queue names, queue
  prefix, enabled flag, and observable count placeholders.
- `LOG_FORMAT=json` starts the server with structured JSON logs.

## Data Contract

- API fields use `camelCase`; database-facing fixtures may use `snake_case`.
- All public IDs are UUID strings.
- Date strings are ISO 8601.
- Default agent keys exist: `coordinator`, `requirements`, `architect`,
  `frontend`, `backend`, `test`, `review`, `notification`.
- A session can have multiple events, briefs, tasks, and artifacts.
- `currentTaskBriefId` points to the latest active brief.
- `rag_retrieval_logs.matched_chunk_ids` corresponds to
  `rag_retrieved.metadata.payload.matchedChunks`.

## Event Contract

Validate every event in the canonical fixture:

- Required fields: `id`, `sessionId`, `type`, `content`,
  `metadata.schemaVersion`, `createdAt`.
- Agent events include `fromAgentId`.
- Mention events include non-empty `toAgentIds`.
- Task events include `taskId` or `metadata.payload.taskId`.
- High-risk tool events include `requiresUserConfirmation`.
- `createdAt` is ISO 8601.
- `renderAs` matches the UI rendering expectation for brief, confirmation, RAG,
  review, delivery, artifact, and error cards.

## Runtime Contract

- `AgentRunInput.phase` is one of the v0.1 phases.
- Runtime receives a bounded `ContextPack`, not full chat history.
- `ContextPack.constraints` is explicit.
- `ContextPack.ragSnippets` includes source identifiers.
- `ExpectedRuntimeOutput.kind` equals `RuntimeOutput.kind`.
- Completed results include `output` and `usage`.
- Failed results include `error`.
- Runtime events can be converted into `collaboration_events`.
- Mock runtime supports deterministic scenarios:
  `happy_path`, `needs_user_decision`, `task_failed`, `brief_conflict`.

## UI State Contract

Feed the canonical event fixture into the event store derivation and assert:

- Default view mode is `chat`.
- `WAIT_USER_CONFIRM` has one active confirmation card.
- `EXECUTING` allows pause, message send, and Agent mention.
- `WAIT_USER_DECISION` is derived after an executing constraint/correction
  message pauses the runtime.
- `rag_retrieved` maps to a chat message with `messageType = "rag"`.
- RAG sources appear in at least one of chat messages or agent cards.
- `post_review_completed` maps to `messageType = "review"`.
- `final_delivery_created` maps to `messageType = "delivery"`.
- SSE reconnect backfill does not duplicate already seen events.

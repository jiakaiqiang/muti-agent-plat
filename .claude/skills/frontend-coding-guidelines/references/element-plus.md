# Element Plus 细则

## 组件选型

- 基础交互优先使用 Element Plus 官方组件；只有在业务语义明显不同、或仓库已有统一二次封装时才封装新组件。
- 封装组件时保留 Element Plus 的核心能力，不要为了简化而砍掉常用插槽、事件或属性透传。
- 同一类交互保持一致：按钮、弹窗、表单校验、分页、消息反馈不要在同一项目里混多套模式。

## 表单

- 使用 `el-form`、`el-form-item` 管理结构和校验，不手写平铺校验流程替代其能力。
- 表单模型保持单一来源，避免 `props`、本地 `ref`、`v-model` 三处重复维护。
- 校验规则贴近字段定义，异步校验明确 loading、失败提示和重复触发控制。
- 搜索表单、弹窗表单、详情表单按场景拆分，不写一个巨型万能表单组件。

## 表格

- 使用 `el-table` 时先明确数据量、分页方式、固定列、选择列和操作列职责。
- 列渲染复杂时抽成具名插槽或小组件，不在单个 `template` 里堆大量条件判断。
- 表格的 loading、empty、error、selection、pagination 状态要成体系出现，不只处理成功态。
- 表格工具栏、筛选区、分页区布局保持一致，不同页面不要各写一套。

## 弹窗与抽屉

- 短流程、确认类操作优先 `ElMessageBox` 或轻量 `el-dialog`；复杂编辑流程再使用完整弹窗或抽屉。
- 弹窗打开时初始化需要的数据，关闭时回收临时状态、校验状态和异步副作用。
- 确认按钮在提交中必须禁用或显示 loading，避免重复提交。

## 反馈组件

- 瞬时结果使用 `ElMessage`，需要更高权重的系统反馈使用 `ElNotification`，破坏性确认使用 `ElMessageBox`。
- 不对每个分支都弹消息；优先在真正需要用户关注的节点反馈。
- 同一类错误反馈文案和时机保持一致，不同页面不要随意变体。

## 样式与主题

- 优先使用 Element Plus 暴露的 token、size、type、CSS 变量和主题能力。
- 主题改动先走全局配置，再考虑局部覆盖；避免直接改组件内部结构样式。
- 若项目已有统一表格、表单、按钮间距规范，优先抽成业务容器类或二次封装组件承载。

## 常见误区

- 不要把业务状态塞进组件私有属性里规避数据流设计问题。
- 不要为了一点视觉差异复制一套官方组件结构。
- 不要在页面里直接散落大量 `ElMessage`、`ElLoading` 调用；复杂流程应收拢到动作函数或服务层。

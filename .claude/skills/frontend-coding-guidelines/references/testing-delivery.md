# 测试与交付细则

## 测试策略

- 优先测试用户能观察到的行为，而不是实现细节。
- 纯函数测输入输出，composable 测状态流转，组件测渲染与交互，页面测关键业务路径。
- 新增逻辑分支、状态机、表单校验、路由守卫或数据转换时补对应测试。
- 如果仓库没有测试基础设施，至少执行手工验证并说明覆盖范围。

## Vue 测试重点

- 组件测试关注 props、slots、emits、条件渲染和异步状态变化。
- composable 测试关注初始化状态、副作用触发、清理逻辑和异常分支。
- Pinia 测试关注 action 行为、getter 派生和边界输入。
- 路由相关测试关注参数变化、导航守卫、重定向和页面状态同步。

## Element Plus 测试重点

- 表单测试关注校验规则、重置行为、禁用态、提交中态和错误提示展示。
- 表格测试关注列渲染、空态、加载态、分页、选择状态和操作列事件。
- 弹窗测试关注打开关闭、确认取消、关闭前校验和状态回收。
- 消息反馈测试关注 `ElMessage`、`ElMessageBox`、`ElNotification` 的触发时机与异常分支。

## 自检清单

- 类型检查通过，无新增类型逃逸。
- lint 通过，无明显死代码、未使用导入、命名冲突或风格回退。
- 构建通过，无打包错误、路由错误或静态资源引用错误。
- 关键交互已验证：加载、空态、错误态、成功态、禁用态。
- Element Plus 关键组件已验证：表单、表格、分页、弹窗、抽屉、下拉选择等核心路径无明显行为偏差。
- 样式已检查：桌面与移动端至少各看一遍，不出现明显溢出、抖动或遮挡。
- 可访问性已检查：键盘可达、焦点可见、表单和按钮语义正确。

## 交付说明

- 总结修改范围：改了哪些页面、组件、composable、store、服务或样式。
- 说明关键权衡：为什么这样拆分、为什么状态放在这里、为什么选这个交互。
- 记录验证结果：执行了哪些命令、哪些场景靠手工验证、哪些仍未覆盖。
- 如果存在后续工作，明确指出而不是隐含留坑。

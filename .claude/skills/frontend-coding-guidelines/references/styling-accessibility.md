# 样式与可访问性细则

## 样式

- 优先复用项目现有样式体系：设计 token、CSS 变量、Element Plus 主题变量、Tailwind 约定或预处理器体系。
- 样式名表达语义或角色，不表达临时视觉结果。
- 避免魔法数字；间距、字号、圆角、阴影优先走统一尺度。
- 不为一个局部视觉问题破坏整体布局约束；先找容器边界和布局原因。
- 响应式设计优先移动端和窄屏可用性，再扩展到大屏。

## Vue 样式约定

- 先遵循仓库既有样式方案：`scoped CSS`、CSS Modules、Tailwind、UnoCSS 或 Element Plus 主题覆盖方案。
- 使用 `scoped` 时注意选择器范围，不依赖深层级穿透解决结构问题。
- 样式逻辑复杂时，优先通过状态类名、CSS 变量或组合 class 控制，而不是内联样式堆判断。
- 需要跨组件复用时，抽离公共变量、mixins、utility classes 或主题 token。

## Element Plus 样式约定

- 先使用 Element Plus 的 size、type、status、slot 和现成布局能力，不先写覆盖样式。
- 主题级改动优先走 token、`ConfigProvider` 或全局变量，不在单页面用深选择器硬改默认皮肤。
- 局部覆盖样式时，先给业务容器加稳定 class，再在容器内定向覆盖，避免全局污染。
- 不依赖脆弱的内部 DOM 结构选择器；如果必须覆盖，控制范围并写明原因。

## 布局

- 明确容器职责：父级控制布局，子级控制内容展示。
- 尽量减少无语义包裹层；只有为布局、定位或语义需要时才新增容器。
- 列表、表格、表单要先考虑窄屏折叠、滚动与截断策略。

## 可访问性

- 优先使用原生语义标签，例如 `button`、`label`、`nav`、`main`，不要用 `div` 模拟交互。
- 交互元素必须可键盘访问，并保持清晰焦点状态。
- 图标按钮提供可访问名称；图片提供合适的 `alt` 或明确标记为装饰。
- 表单控件绑定标签、错误提示和必填信息；使用 `el-form` 时不要因为视觉封装而丢失语义关系。
- 颜色不是唯一信息通道；错误、成功、警告状态要有文本或图标辅助。

## 交互反馈

- 用户动作后尽快给反馈：loading、success、error、disabled、skeleton 至少选其一。
- 避免无反馈的异步操作，也避免“全局大转圈”替代局部反馈。
- 空态文案说明当前没有什么、为什么没有、下一步能做什么。

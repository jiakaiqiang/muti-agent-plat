# 11 Delivery Memory Practice 第四阶段交付记忆与复盘机制

> **工程化定位** · 控制面: Entropy Management · 权威来源: 本文件 · 修改触发: 长闭环自身演化

## 目的

第四阶段不是写代码，不是新增系统功能，也不是把交付记录做成数据库。

第四阶段的目标是：

```text
建立一套工程化的交付记忆与复盘机制，
让每次 Agent 协作交付后的经验能够被筛选、沉淀、复用和清理。
```

第一阶段定义规程。

第二阶段让现有协作流程对齐规程。

第三阶段让每个 Agent 有明确工作协议。

第四阶段解决：

```text
一次交付结束后，哪些东西应该留下来，哪些东西不应该留下来。
```

## 第四阶段要解决的问题

如果没有交付记忆机制，本项目会出现：

```text
每次都重新理解项目
相同失败反复出现
设计决策无法追溯
用户偏好只留在聊天记录里
经验被写散在多个文档里
长期记忆混入一次性噪音
旧经验过期后无人清理
```

第四阶段不是简单“多写总结”，而是建立记忆筛选和复盘规则。

## 交付记忆的分类

5 类记忆的权威定义见 [`08-delivery-memory.md`](./08-delivery-memory.md)。本文件不重复列出标题，下面展开各类的"应该沉淀 / 不应该沉淀 / 复盘问题"。

```text
1. Project Knowledge      项目知识
2. Design Decisions       设计决策
3. Failure Patterns       失败模式
4. Verification Evidence  验收证据经验
5. User Preferences       用户偏好
```

## 1. Project Knowledge 项目知识

### 应该沉淀

```text
能力域与代码范围的稳定关系
模块边界
常用入口文件
重要配置位置
跨模块依赖关系
现有约束
```

### 不应该沉淀

```text
一次性调试路径
临时日志
某次运行的随机 ID
只对当前需求有意义的细节
未确认的猜测
```

### 复盘问题

```text
这次交付是否发现新的能力域？
是否明确了新的模块边界？
是否发现已有文档和实际项目不一致？
后续 Agent 是否能因此更快定位相关代码？
```

## 2. Design Decisions 设计决策

### 应该沉淀

```text
被采用的方案
被放弃的方案
选择原因
约束条件
适用范围
未来可能需要重新评估的条件
```

### 不应该沉淀

```text
没有比较过的主观偏好
未经 Review 的临时方案
为了当前任务临时绕开的设计
已经被否决但没有复用价值的细节
```

### 复盘问题

```text
这个设计决策以后还会影响类似需求吗？
这个决策依赖哪些前提？
如果前提变化，是否需要重新评估？
```

## 3. Failure Patterns 失败模式

### 应该沉淀

```text
重复出现的失败类型
导致返工的常见原因
上下文缺失模式
Agent 越权模式
工具权限误用模式
验收证据不足模式
```

### 不应该沉淀

```text
偶发环境问题
一次性拼写错误
无复现价值的临时失败
没有根因的失败描述
```

### 复盘问题

```text
这次失败以后可能再次发生吗？
它属于需求、设计、实现、验证、权限、上下文中的哪一类？
是否需要更新 Feedback Loop 或 Context Protocol？
```

## 4. Verification Evidence 验收证据经验

这里的 Verification 不是测试中心，而是“哪些证据能说明交付满足需求”。

### 应该沉淀

```text
对某类需求有效的验收方式
能证明契约被满足的证据类型
评审认为可信的证据
无效或误导性的验证方式
```

### 不应该沉淀

```text
单次命令输出全文
无解释的截图
无法复用的测试日志
只证明运行过但不能证明需求满足的证据
```

### 复盘问题

```text
这次哪些证据真正证明了验收标准？
哪些证据只是噪音？
以后类似需求应该优先收集什么证据？
```

## 5. User Preferences 用户偏好

### 应该沉淀

```text
用户明确表达的长期偏好
用户确认过的工程约束
用户反复强调的交付格式
用户明确反对的行为
```

### 不应该沉淀

```text
用户一时情绪
只针对本次任务的选择
没有明确确认的推测
与项目无关的偏好
```

### 复盘问题

```text
这个偏好是否长期有效？
用户是否明确表达过？
是否会影响后续 Agent 协作？
是否需要在下次需求阶段主动考虑？
```

## 交付复盘流程

每次 Delivery 阶段结束后，必须进行一次轻量复盘。

复盘按以下顺序：

```text
1. 回看 Intent Contract：需求是否被满足？
2. 回看 Design Plan：哪些设计决策值得保留？
3. 回看 Feedback Loop：是否发生返工，原因是什么？
4. 回看 Tool Governance：是否出现权限或工具问题？
5. 回看 Verification Evidence：哪些证据最有效？
6. 回看 Review Report：哪些风险仍需跟进？
7. 判断哪些内容进入 Delivery Memory。
```

## 记忆写入规则

写入规则的权威定义见 [`08-delivery-memory.md`](./08-delivery-memory.md)。本文件不重复定义，下面只补充复盘视角的四个判断问句：

```text
1. 是否可复用？
2. 是否已确认？
3. 是否有明确适用范围？
4. 是否不会污染后续 Agent 判断？
```

只有四项都满足，才进入长期记忆。

否则只保留在本次交付记录中。

## 记忆去重规则

沉淀前必须检查：

```text
是否已有相同知识？
是否只是已有知识的换一种说法？
是否应该更新旧知识，而不是新增一条？
是否旧知识已经过期？
```

## 记忆过期规则

以下内容需要标记为可能过期：

```text
依赖具体版本的经验
依赖当前目录结构的路径
依赖当前 runtime 配置的结论
依赖临时业务目标的约束
```

过期记忆不能直接删除，应该先标记：

```text
status: active | stale | superseded
```

## 每次交付的复盘记录格式

建议每次交付后生成：

```text
交付编号：
需求摘要：
是否完成：
主要设计决策：
发生的返工：
失败模式：
有效验收证据：
用户偏好：
沉淀为长期记忆的内容：
只保留在本次记录的内容：
需要后续清理或复查的内容：
```

## 第四阶段不做什么

第四阶段不做：

```text
不新增数据库
不新增系统功能
不新增记忆 API
不做自动知识库
不把所有交付日志都写入长期记忆
```

第四阶段只做：

```text
交付复盘规则
记忆分类规则
记忆写入规则
记忆去重规则
记忆过期规则
知识污染防护规则
```

## 完成标准

第四阶段完成的标志是：

```text
每次交付结束后，都知道应该复盘什么。
每类经验都知道是否应该沉淀。
长期记忆有进入标准。
一次性噪音不会污染长期知识。
失败模式能反向改进前面的规程。
用户偏好能被明确确认后复用。
```

完成第四阶段后，才适合进入第五阶段：持续治理与演进。

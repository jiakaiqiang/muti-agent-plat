# 08 Delivery Memory 交付记忆

> **工程化定位** · 控制面: Entropy Management · 权威来源: 本文件 · 修改触发: 长闭环自身演化

## 目的

规定一次交付结束后，哪些经验需要沉淀为项目知识。

## 需要沉淀的内容

### 项目知识

- 本次需求属于哪个能力域。
- 涉及哪些代码或文档范围。
- 哪些模块边界被确认。

### 设计决策

- 采用了什么方案。
- 放弃了什么方案。
- 为什么这样选择。

### 失败模式

- 哪些问题导致返工。
- 哪些上下文不足。
- 哪些权限或工具边界需要注意。

### 验收经验

- 哪些证据能有效证明交付满足需求。
- 哪些验证方式不可靠。

### 用户偏好

- 用户明确表达的偏好。
- 用户确认过的约束。

## 写入规则

- 只有对后续任务有复用价值的内容才沉淀。
- 不把一次性细节写成长期知识。
- 不重复写入已有知识。
- 不确定是否长期有效时，只留在本次交付记录中。

## 退出条件

- 最终交付说明了哪些内容被沉淀。
- 沉淀内容能帮助后续 Agent 更快理解项目。
- 没有把噪音、临时决策或过期信息写成长期记忆。

## 沉淀触发与字段对接

本节定义沉淀动作的触发时机与字段来源。对应闭环模型中的边 ⑤（Feedback → Entropy）。

### 触发时机

沉淀动作必须在以下时机执行，不允许仅在"任务结束"统一处理：

- 任意一次返工发生时：触发"失败模式"沉淀候选。
- Review 给出 `rework` 或 `fail` 决策时：触发"失败模式" + "设计决策"沉淀候选。
- Delivery 阶段完成时：触发全部 5 类的最终筛选。
- Human Intervention 完成时：触发"用户偏好"沉淀候选。

### 字段映射

来自 `../feedback-loop/07-feedback-loop.md` 返工信号的字段映射到本文件的 5 类记忆：

| 07 对外信号字段 | 映射到 |
| --- | --- |
| 失败分类 | 失败模式 |
| 上下文缺口 | 失败模式（"哪些上下文不足"项） |
| 待修正产物 | 设计决策（被否定的方案） |
| 证据 | 验收经验（"哪些验证方式不可靠"项） |
| 期望修正结果 | 失败模式（"哪些权限或工具边界需要注意"项） |

### 沉淀候选与正式沉淀

- 候选：触发时机发生时先记为"候选"，不直接成为长期记忆。
- 正式沉淀：经 Delivery Agent 按"写入规则"筛选后，才进入长期记忆。
- 候选保留期：本次交付周期内有效，未被正式沉淀的候选随交付归档而清理。

### 硬规则

- 沉淀候选不得跳过字段映射表中的来源字段，缺字段时不得登记候选。
- 同一失败模式在同一交付内只允许沉淀一次，不允许多次返工产生重复记忆。
- 沉淀内容必须可被 `12-continuous-governance.md` 检索，作为漂移检测和回写动作的输入（对应边 ⑥⑦⑧）。

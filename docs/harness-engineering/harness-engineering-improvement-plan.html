Harness Engineering / 改进建议与实施方案

从阶段式文档堆叠，重构为四控制面工程体系

日期：2026-06-27 · 状态：建议方案 · 范围：.claude/harness-engineering/ 文档体系
目标：降低冗余，强化约束，让 Harness Engineering 作为 AI 系统工程化规程稳定演进。

1. 结论摘要

当前 Harness Engineering 的内容方向是正确的，但信息架构仍以 01-12 阶段编号为主线，导致核心规则、落地方案、运行时对齐、复盘治理混在同一层级。建议将主结构调整为四个控制面： 上下文工程、架构约束、反馈循环、熵管理。

4 个控制面 作为 Harness Engineering 的一级架构
01-12 降级 从主目录主线降级为来源文档或 legacy
保留旧引用 第一阶段不删除旧文件，降低破坏性
先逻辑后物理 先建立新入口，再逐步迁移内容

规则与实践存在重复感 高 08-delivery-memory 与 11-delivery-memory-practice、03-agent-role 与 10-agent-working-protocol 职责接近。 后续维护时容易多处修改同一规则，产生漂移。

建议将 Harness Engineering 的一级目录改为控制面，而不是阶段编号。

context-engineering/ 管理需求表达、上下文构造、提示词上下文、阶段输入。 01-intent-contract、02-context-protocol、prompt-context/*

5. 实施方案

阶段 A：建立新信息架构，不移动旧文件

  1. 重写 .claude/harness-engineering/README.md，将四控制面作为主入口。
  2. 新增四个控制面目录及 README：context-engineering、architecture-constraints、feedback-loop、entropy-management。
  3. 每个控制面 README 只做职责定义和旧文件映射，不复制旧规则全文。
  4. 更新 alignment/README.md，明确它是运行时映射层，不是独立控制面。

阶段 B：去重和权威规则收敛

  1. 为每条规则标记唯一权威位置，例如“交付记忆写入规则”归 entropy-management。
  2. 将 03 与 10 的重复内容拆分：03 保留边界原则，10 保留执行手册。
  3. 将 08 与 11 的重复内容拆分：08 保留原则，11 保留复盘流程。
  4. 在旧文件头部增加“权威来源 / 被哪个控制面管理”的说明。

阶段 C：运行时契约对齐

  1. 更新 artifacts-alignment.md：补充 html ArtifactType，并区分 storage type、harnessArtifactType、deliveryFormat。
  2. 更新 orchestrator-alignment.md：将 ExecutionOutcome 归入反馈循环状态机。
  3. 更新 prompt-context：将其归入上下文工程，不再作为独立 Phase 3 主线。
  4. 更新 capability-binding：将其归入架构约束和反馈循环交界处。

阶段 D：物理迁移和清理

  1. 在新结构稳定后，移动旧文件到控制面目录或 legacy/。
  2. 批量修正相对链接。
  3. 删除重复段落，保留引用关系。
  4. 执行一次文档审计，确保没有两个文件同时定义同一条权威规则。

7. 风险与控制

风险 控制方式
一次性移动大量文件导致链接断裂。 第一阶段只新增入口和映射，不移动旧文件；第二阶段再做物理迁移。
新旧体系并存导致更混乱。 README 明确“新结构是权威入口，旧 01-12 是来源文档”。
四控制面边界仍然模糊。 为每个目录写入“负责 / 不负责 / 权威规则类型”。
文档格式规则与 artifact 契约继续冲突。 优先更新 artifacts-alignment，加入 HTML 和 deliveryFormat 概念。

8. 验收标准

  • Harness Engineering README 首屏能说明四控制面，而不是先展示 01-12 阶段列表。
  • 每个控制面都有明确的“负责 / 不负责 / 管理哪些旧文件”。
  • 每条核心规则能找到唯一权威位置。
  • alignment、prompt-context、capability-binding、delivery-memory 不再作为独立思想轴解释。
  • 可阅读交付物的 artifact 契约与默认 HTML 规则一致。
  • 旧文件路径在迁移前仍可访问，不破坏现有引用。

9. 推荐下一步

  1. 先执行阶段 A：新增四控制面目录和 README，重写总 README。
  2. 再执行阶段 C 的关键契约修正：优先更新 artifacts-alignment.md。
  3. 最后再做阶段 B/D 的去重和物理迁移。
建议不要一开始就移动 01-12 文件。先让新信息架构成为权威入口，等引用关系稳定后再做目录迁移。
